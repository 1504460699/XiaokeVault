## 一、未完成的设计承诺（最该补，有明确缺口）

### #1 扫描未知扩展名提示 ⭐⭐⭐
- **现状**：后端 `ScanReport.unknown_extensions` 已统计（实测有 dts/url/html/woff 等约 80 个），但前端**完全没有展示**
- **缺口**：M7 验收标准明确要求"扫描发现的未知扩展名能一键转为新类型"
- **价值**：这是"类型可扩充"闭环的最后一块——用户扫描后看到"发现 3 个 .webm"，点击即可创建 video 类型
- **工作量**：小（TopBar 扫描报告旁加提示徽标 + 点击跳转 TypeSettings 预填扩展名）
- **验收标准**：
  - [ ] 扫描后顶栏显示"发现 N 个未知类型"徽标
  - [ ] 点击徽标打开类型管理，未知扩展名可一键创建为新类型
  - [ ] 创建后提示重新分类

### #2 fbx 加载 ⭐
- **现状**：ModelViewer 里 `fbx 加载需要 fflate 依赖（M6 待补）`，库里 fbx 有 381 个（13_3D模型 209 + 其他 172）
- **缺口**：3D 预览覆盖了 obj/glb/dae/blend，唯独 fbx 缺
- **价值**：中等——补全 3D 格式覆盖
- **工作量**：小（装 fflate + 接 FBXLoader）
- **验收标准**：
  - [ ] 装 fflate 依赖
  - [ ] ModelViewer 支持 fbx 加载（旋转/缩放正常）
  - [ ] 抽查库里 fbx 文件能显示

### #3 GIF 逐帧播放 + 音频波形 ⭐
- **现状**：GIF 用 `<img>` 显示首帧（无逐帧控制），音频用原生 `<audio>`（无波形图）
- **设计承诺**：§5.3「GifPlayer 点开逐帧播放，可调速度」「audio 波形缩略图」
- **价值**：中等——游戏素材里动画帧/音效波形是判断质量的关键
- **工作量**：中（GIF 拆帧播放器 + 音频 Web Audio API 提取波形）
- **验收标准**：
  - [ ] GIF 预览支持逐帧播放/暂停/调速
  - [ ] 音频预览显示波形图（可选保留播放控件）
  - [ ] 缩略图墙的 GIF 显示首帧、音频显示波形缩略

## 二、实测发现的功能薄弱点（影响日常使用）

### #4 文件级勾选不持久化读取 ⭐⭐⭐
- **现状**：文件级勾选**写入**了 DB（selections 表 scope='file'），但进包时 `selectedFileIds` 被 `resetFileSelections` 清空，**不回填**
- **后果**：勾了某文件，离开包再回来，勾选状态在 UI 上**丢失**（虽然 DB 里有，导出仍正确，但用户看不到会困惑）
- **价值**：高——明显的体验缺陷
- **工作量**：中（后端加 `get_package_selected_files(project_id, pkg_id)` command，进包时回填 selectedFileIds）
- **验收标准**：
  - [ ] 后端加查询单文件勾选的 command
  - [ ] 进包时从 DB 回填 selectedFileIds
  - [ ] 勾选文件 → 离开包 → 回来，勾选框仍显示已选

### #5 全局跨包搜索 ⭐⭐⭐
- **现状**：搜索只在当前包内（FileGrid 局部过滤）
- **场景**：想找所有"explosion"素材，但分散在多个包——当前得逐个包翻
- **价值**：高（用户在 M4 时提过"几百文件里找太费劲"，跨包是更大痛点）
- **工作量**：中（M1 设计时就预留了 `search_files` command 但没实现）
- **验收标准**：
  - [ ] 后端 `search_files(query, filters)` command（按文件名+类型跨包搜索）
  - [ ] 顶栏搜索框触发全局搜索，结果在独立视图展示
  - [ ] 搜索结果可勾选、可预览
  - [ ] 支持类型/分类过滤

### #6 增量扫描自动触发 ⭐⭐ ✅
- **现状**：notify crate 监听库根目录，3 秒防抖后触发增量扫描；前端监听 `library://auto-scanned` 自动刷新
- **关键修复**：watcher 必须通过 `app.manage()` 保活，否则 `RecommendedWatcher` 被 drop 后 OS 级监听立即停止（初版 bug：返回值被 `let _ =` 丢弃导致监听不生效）
- **验收标准**：
  - [x] 监听库根目录文件变化
  - [x] 检测到新增/修改时自动增量扫描
  - [x] 扫描进度提示（TopBar "⏳ 自动扫描中…"）

## 三、规模化后的稳健性（库会越来越大）

### #7 缩略图生成（当前直接加载原图） ⭐⭐
- **现状**：缩略图墙用 convertFileSrc 直接加载**原始大图**（有的几 MB）
- **后果**：02_地牢RPG 那种 2.8 万文件的包，滚动时加载大量大图会卡
- **设计原方案**：§4.4 后端生成小尺寸缩略图缓存，但现在没做
- **价值**：高（性能，大库必备）
- **工作量**：中大（Rust image crate 生成缩略图 + LRU 缓存 + 前端改用缩略图 URL）
- **验收标准**：
  - [ ] 后端 `get_thumbnail(file_id)` 生成 200px 缩略图，缓存到 app data
  - [ ] 缩略图墙用缩略图 URL 替代原图
  - [ ] 大包（2.8万文件）滚动流畅
  - [ ] 缓存设上限，LRU 清理

### #8 哈希去重（M5 只做了 zip_extracted） ⭐⭐
- **现状**：dedup 只检测"压缩包+解压目录"，`likely_backup` 和 `hash` 检测**写了函数没调用**
- **缺口**：README 提到的"明显备份目录"（如 DungeonCrawl 完整版 vs Stone_Soup_Full）没被检测
- **价值**：中
- **工作量**：中（likely_backup 名字相似度 + hash 按 sha256 比对）
- **验收标准**：
  - [ ] likely_backup 检测同名/近名包共存（带相似度评分）
  - [ ] hash 检测内容完全相同的文件（按需算 sha256）
  - [ ] DedupPanel 分类展示三种重复（zip_extracted/likely_backup/hash）

## 四、工程质量与发布（从"能用"到"好用"）

- **工作量**：小（网络好时挂代理重跑 `npm run tauri build`，或手动补全工具包）
- **验收标准**：
  - [ ] 生成 nsis 安装包（.exe 安装程序）
  - [ ] 安装后有开始菜单快捷方式
  - [ ] 控制面板可卸载

### #10 代码分块（three.js 导致 bundle 841KB） ⭐
- **现状**：构建警告 chunks > 500KB（主要是 three.js，主 bundle 841KB / gzip 230KB）
- **价值**：低（功能不影响，仅首屏加载稍慢）
- **工作量**：小（vite manualChunks 把 three 单独分包 + ModelViewer 动态 import 懒加载）
- **验收标准**：
  - [ ] three.js 独立分包，主 bundle < 500KB
  - [ ] ModelViewer 按需加载（进 3D 预览才加载 three）
  - [ ] 无 > 500KB chunk 警告

### #11 错误处理与日志 ⭐⭐
- **现状**：很多 `map_err(|e| e.to_string())`，错误信息对用户不友好；前端部分操作静默失败（如 M7 之前的 upsert）
- **价值**：中（出问题时方便定位，提升健壮性）
- **工作量**：中（统一错误类型 + 友好提示 + 关键路径日志）
- **验收标准**：
  - [ ] 后端关键操作返回结构化错误（而非裸字符串）
  - [ ] 前端所有 invoke 调用有 try/catch + 用户可见提示
  - [ ] 用户能看懂的中文错误提示

### #12 测试覆盖 ⭐
- **现状**：只有 M1 的 2 个集成测试（scanner/indexer），M2-M7 零测试
- **价值**：中（防回归，重构时有底气）
- **工作量**：大（selection/exporter/dedup 的单元测试 + 端到端测试）
- **验收标准**：
  - [ ] selection 模块测试（勾选/汇总逻辑）
  - [ ] exporter 模块测试（导出结构/credits/manifest 正确性）
